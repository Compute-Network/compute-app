# v0.4.2 — oMLX backend integration (Mac MLX route)

## Goal

A user sending a request for model `qwen-3.6` gets:
- **Mac** (Apple Silicon): served by oMLX running the MLX variant (`mlx-community/Qwen3.6-35B-A3B-4bit`). No manual install, no manual model download, no backend flag.
- **Linux / Windows**: served by llama-server running the GGUF variant (`unsloth/Qwen3.6-35B-A3B-GGUF` / `Qwen3.6-35B-A3B-UD-Q4_K_M.gguf`). Same — auto.

The orchestrator sees one model id (`qwen-3.6`); the daemon picks the right format for its host.

## What v0.4.1 already landed

- `ModelDefinition::mlx: Option<MlxVariant>` in `crates/compute-network/src/models.rs` — catalog carries both MLX and GGUF pointers per model.
- `ModelDefinition::preferred_format_here() -> Option<BackendFormat>` — cfg-gated; returns `Mlx` on `macos + aarch64` when the model has an MLX variant, else `Gguf`.
- `BackendFormat` enum (`Mlx | Gguf`), serde-friendly for heartbeat / assignment payloads.
- `qwen-3.6` catalog entry with both paths wired.

Nothing runtime-facing has shifted: the daemon today still downloads and serves GGUFs through llama-server. The catalog just *knows* about the MLX route.

## v0.4.2 scope

### 1. oMLX install, driven by `install.sh`

Current install.sh ships llama-server + llama_stage_* via a pre-built tarball. For oMLX on Macs, add a conditional post-install:

```sh
# In compute-website/public/install.sh, after the main tarball is unpacked,
# and only when OS=darwin:
if [ "$OS" = "darwin" ]; then
  if ! command -v omlx >/dev/null 2>&1; then
    if command -v brew >/dev/null 2>&1; then
      spin "Installing oMLX (MLX inference server)" \
        "brew tap jundot/omlx https://github.com/jundot/omlx 2>&1 >/dev/null && brew install omlx"
    else
      echo "  ${DIM}Note: oMLX requires Homebrew. MLX-format models will fall back to GGUF.${RESET}"
    fi
  fi
fi
```

**Why brew**: oMLX is a Python + FastAPI package. `pip install` requires a suitable Python; `brew install` handles the Python toolchain and writes a launcher to `/opt/homebrew/bin/omlx`. The install.sh's `✓ Verified` block should probe for `omlx --help` and surface the version.

Acceptable degradation: if brew isn't present / the install fails, the daemon falls back to GGUF via llama-server. Log a warn so the user knows they're not on the faster path.

### 2. Daemon-side MLX backend

**New file**: `crates/compute-daemon/src/inference/mlx.rs`

```rust
pub struct OmlxManager {
    port: u16,
    child: Option<Child>,
    loaded_model_folder: Option<PathBuf>,
    model_cache_mlx_dir: PathBuf,  // ~/.compute/models/mlx
}
```

Lifecycle mirrors `InferenceManager` (llama-server wrapper):
- `ensure_downloaded(repo_id, folder)` — HuggingFace snapshot download (prefer `hf_hub_download` via a small helper, or shell out to `huggingface-cli`) into `<cache>/mlx/<folder>/`.
- `start(folder_path)` — spawns `omlx serve --model-dir <parent_of_folder> --port <port>` as child; waits on `/v1/models` for readiness.
- `stop()` — SIGTERM child, 5s grace, SIGKILL.
- `is_alive()` — `kill(pid, 0)`.
- `proxy_openai_request(...)` — forward `/v1/chat/completions` to `localhost:<port>/v1/chat/completions`.

**Port allocation**: reuse llama-server's port convention but on +1. llama-server is `8090`; oMLX is `8091`. Both can run simultaneously (they hold different models, or one is parked per the v0.4.1 keep-warm pattern).

### 3. Runtime router (`runtime.rs`)

When an assignment arrives with a model id whose `preferred_format_here() == Mlx`:
- Route to the `OmlxManager` instead of `InferenceManager`.
- Park the llama-server manager (already done per v0.4.1 keep-warm semantics — just branch based on format).

In the relay layer (`relay.rs:handle_inference_request`), add a selection step *before* the existing `gateway_client / stage_client / llama-server` ladder:

```rust
let catalog = ModelCatalog::default_catalog();
if let Some(model_def) = catalog.find(&requested_model) {
    if model_def.preferred_format_here() == Some(BackendFormat::Mlx) {
        if let Some(mlx) = mlx_manager.as_ref() {
            return proxy_to_omlx(mlx, request).await;
        }
        // fall through to GGUF path (warn once)
    }
}
// existing gateway_client → stage_client → llama-server ladder
```

### 4. Heartbeat & orchestrator placement

Daemon heartbeat (`compute-daemon/src/relay.rs` — the HeartbeatPayload struct) gains:

```rust
/// Backends this node can actually serve, after inspecting installed
/// binaries at startup. `mlx` is only ever true on macos + aarch64 with
/// oMLX installed. `gguf` is essentially always true (llama-server ships
/// in the main tarball). Orchestrator reads this to avoid routing an MLX
/// assignment to a Linux node.
#[serde(default)]
pub supported_backends: Vec<BackendFormat>,
```

Orchestrator-side (`compute-orchestrator/src/services/placement.ts`):
- When selecting nodes for an assignment, filter by `supported_backends`.
- For single-node models with an MLX variant, strongly prefer Mac nodes; fall back to GGUF-capable nodes if no Mac is available.

### 5. Model download resolver

New helper in `crates/compute-daemon/src/inference/stage_artifacts.rs`:

```rust
/// HF snapshot download of an MLX model folder into the cache.
pub async fn ensure_mlx_folder(
    cache_dir: &Path,        // ~/.compute/models
    repo_id: &str,           // "mlx-community/Qwen3.6-35B-A3B-4bit"
    folder: &str,            // "Qwen3.6-35B-A3B-4bit"
) -> Result<PathBuf> {
    let dest = cache_dir.join("mlx").join(folder);
    if dest.join("config.json").exists() {
        return Ok(dest);  // already cached
    }
    // Download via huggingface-cli (bundled with omlx via brew) OR a
    // pure-Rust huggingface-hub client. Prefer CLI for simplicity.
    Command::new("huggingface-cli")
        .args(["download", repo_id, "--local-dir"])
        .arg(&dest)
        .spawn()? ... etc.
    Ok(dest)
}
```

### 6. Config — one new field, defaulted

Add `experimental.omlx_port: u16 = 8091` for deterministic binding. No user action required.

## Non-goals for v0.4.2

- Split inference for MLX models. oMLX is single-node; `qwen-3.6.recommended_stages = 1` enforces this.
- MLX draft models for spec decoding. `qwen-3.6` has no `draft_model_id` yet. DFlash MLX support is a separate spike.
- Windows WSL / Linux MLX support. MLX is Apple-silicon only.

## Validation plan

1. Fresh Mac install: `curl install.sh | sh` → brew installs oMLX, binary lands in `~/.compute/bin`.
2. `compute` launches, daemon heartbeats with `supported_backends: [mlx, gguf]`.
3. Orchestrator assigns `qwen-3.6` to the Mac. Daemon resolves `preferred_format_here = Mlx`. Downloads MLX folder on first assignment (~20 GB, ~5 min on fast pipe).
4. Spawns `omlx serve --port 8091`; waits for `/v1/models` ready.
5. Routes chat completion via oMLX. Measures tps; compare to the same prompt on llama-server-GGUF for the same model.
6. Fresh Linux install: same flow, daemon heartbeats `supported_backends: [gguf]`, orchestrator avoids sending MLX-preferring assignments, daemon falls through to GGUF+llama-server.
7. Mixed cluster: orchestrator prefers Mac for `qwen-3.6` even if a Linux GPU node is available, because MLX path is faster.

## Open questions

- **Who owns the brew tap?** Currently `jundot/omlx`. If oMLX disappears or changes license, we need a fallback. Mitigation: keep GGUF+llama-server as the first-class default; MLX is a bonus path.
- **Disk space on Macs**: 20 GB for Qwen 3.6 MLX + 22 GB for Qwen 3.6 GGUF = 42 GB if a Mac ends up downloading both. Solution: Mac nodes only download the format their `preferred_format_here()` returns.
- **oMLX continuous batching**: oMLX supports batching. Daemon should proxy multiple concurrent requests rather than serializing them — wire into the existing relay concurrency model.

## Status after v0.4.2

- User runs `compute` on any OS with any supported model id; the correct backend fires automatically.
- Install.sh is the only touch point; no `compute config set` incantations.
- The keep-warm pattern from v0.4.1 extends naturally: `OmlxManager` parks when llama-server or gateway takes over, resumes on flip-back.
